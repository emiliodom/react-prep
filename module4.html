1. Core Concepts

Controlled Inputs: In HTML, inputs manage their own state. In React, we hijack that. value={state} and onChange={setState}. The React state becomes the "single source of truth".

Context API: Solves "Prop Drilling" (passing data through 5 layers of components that don't need it).

  • createContext: Creates the data pipe.
  • Provider: Puts data into the pipe at the top.
  • useContext: Extracts data from the pipe at the bottom.

1.1 Deep Dive: Controlled vs Uncontrolled

While "Controlled" (React state drives input) is the standard, "Uncontrolled" (DOM drives input, React reads it via Ref) is valid for simple forms or integrating with non-React libraries. However, for complex validation, Controlled is king.

1.2 Industry Standard: Form Libraries

Building forms from scratch is painful. In the real world, we use libraries like React Hook Form or Formik. They handle validation, error messages, and "touched" states (knowing if a user has clicked a field) much better than manual state.

For this module, build it manually to understand the pain. In Module 5, we might use a library.

2. The Build Challenge: "Login Flow"

Level 1: The Basics

Create a LoginForm component.

  • Fields: Email, Password.
  • Validation:
    • Email must contain "@".
    • Password must be > 6 chars.
    • Disable the "Submit" button if invalid.
  • Show error messages in red below the inputs.

Level 2: Senior Constraints (Context Architecture)

Create an AuthContext.

  • Type Safety: Define a proper interface for your Context Value (e.g., AuthContextType). Do not default the context to null without handling it. (Tip: createContext<AuthContextType | undefined>(undefined)).
  • Custom Hook: Create a useAuth() hook that wraps useContext. It should throw an error if used outside of an AuthProvider. This is a fail-fast mechanism.
  • Logic:
    • Wrap your entire app in AuthProvider.
    • When the LoginForm submits successfully, call login() from the context.
    • Update the Header to show "Welcome, [Name]" and a "Logout" button if the user is logged in.
    • Redirect to the Home page upon login.

Level 3: Nice to Haves

  • Persisted Auth: Save the user's login token (fake it) to localStorage. On app refresh, check storage and auto-login the user if the token exists.
  • Protected Route: Create a wrapper component <RequireAuth>. If a non-logged-in user tries to visit /dashboard, automatically redirect them to /login.

3. The Bug Hunt

üêõ Scenario: Excessive Re-renders

The Setup: Put a console.log("Rendering App") in your root App component.

The Issue: Type into your email input. Does the entire App re-render on every keystroke? If your state is lifted too high, it might.

The Task: Optimize. Ensure that typing in the form only re-renders the form, not the Header or Footer.

4. Test on your own

Verify your State

  • Refresh Check: Log in, then refresh the page. Are you still logged in? (Only if you did Level 3). If not, does the app crash?
  • Redirect Check: If you are already logged in, try to manually visit /login. A smart app should redirect you back to the dashboard because you don't need to login twice.
