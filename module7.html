1. Core Concepts

Unit vs Integration: Unit tests check one function in isolation. Integration tests check how components work together (e.g., clicking a button updates the list).

React Testing Library (RTL): Tests from the user's perspective. It doesn't care about your state variables. It cares if the user can see the text "Success" on the screen.

Vitest: The test runner. It executes your test files. It's a faster, modern replacement for Jest.

1.1 Deep Dive: The Testing Pyramid

Google's testing philosophy suggests:
1. Unit Tests (70%): Fast, cheap, test individual functions.
2. Integration Tests (20%): Test how components interact.
3. E2E Tests (10%): Slow, expensive, test the full app flow (using Playwright/Cypress).

1.2 Industry Standard: Mocking

In a real job, you never hit the real API in a unit test. It's slow and flaky. We use MSW (Mock Service Worker) to intercept network requests at the network level. This is superior to mocking fetch directly because it mimics the browser's behavior more closely.

2. The Setup Challenge

Task: Configure Test Environment

Vite doesn't come with tests by default.

  1. Install vitest, jsdom, and @testing-library/react.
  2. Configure vite.config.ts to support testing.
  3. Add a test script to package.json.

3. The Build Challenge: "Test Coverage"

Level 1: The Basics

Write tests for your previous components.

  1. Unit Test (UserCard):
    • Render the card.
    • Assert that the name and role are displayed.
    • Assert that the "green dot" class is present if isActive is true.
  2. Integration Test (Counter):
    • Render the Counter.
    • Find the "Increment" button and click it (fireEvent or userEvent).
    • Assert that the count text updates to "1".

Level 2: Senior Constraints (Async & Mocking)

  • Async Test (User List):
    • Mock the API call (use vi.spyOn(global, 'fetch') or MSW).
    • Assert that "Loading..." appears first.
    • Assert that user names appear after the promise resolves.
  • Test a Custom Hook: Create a hook useCounter. Use renderHook and act from testing-library to test it in isolation (without a component).
  • Accessibility Test: Use jest-axe (or vitest equivalent) to assert that your component has no accessibility violations.

Level 3: Nice to Haves

  • Custom Render: Create a custom render function that automatically wraps components in your Providers (Theme, Auth, Redux). This saves you from wrapping every single test manually.
  • Snapshot Testing: Add a snapshot test for a static component (like a Footer) to catch accidental UI changes.

4. The Bug Hunt

üêõ Scenario: The "Act" Warning

The Setup: Update a state inside a useEffect or async operation, but assert the result immediately in your test without waiting.

The Issue: React will warn: "An update to Component inside a test was not wrapped in act(...)".

The Task: Fix this using waitFor or findBy queries from React Testing Library.

5. Test on your own

Verify your Tests

  • Breakage Check: Go into your UserCard component and change the name rendering logic (e.g., make it uppercase). Run your tests. Do they fail? (They should). If they pass, your test is useless.
  • Async Check: In your User List test, add a await new Promise(r => setTimeout(r, 100)) inside your mock to simulate network lag. Does your test still pass? (It should, if you used findBy correctly).
