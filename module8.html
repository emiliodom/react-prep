1. Core Concepts

Linting (ESLint): Static analysis. It finds bugs before you run code (e.g., "You defined 'x' but never used it", or "You broke the Rules of Hooks").

Formatting (Prettier): Style enforcement. It ensures everyone's code looks the same (indentation, quotes, semicolons) regardless of who wrote it.

Husky (Git Hooks): Automation. It runs your linter and tests before you commit. If they fail, the commit is blocked. This keeps the repository clean.

1.1 Deep Dive: Static Analysis

Linting isn't just about style; it's about catching logic errors. Rules like react-hooks/exhaustive-deps save you from infinite loops. Rules like no-floating-promises save you from unhandled async errors.

1.2 Industry Standard: Conventional Commits

Professional teams use a standard format for commit messages: feat: add user login, fix: resolve crash on safari. This allows tools to automatically generate changelogs and semantic version numbers.

1.3 Senior Skill: Debugging vs. Console.log

Juniors spam console.log. Seniors use the Debugger.
Learn to use the debugger; statement or VS Code's "Run and Debug" tab. You can pause execution, inspect variables in real-time, and step through the call stack. It is 10x faster than logging and reloading.

2. The Build Challenge: "The Guardrails"

Level 1: The Basics

  1. ESLint & Prettier:
    • Install and configure them to work together.
    • Add a rule: No console.log allowed.
    • Add a rule: Imports must be sorted alphabetically.
  2. Husky & Lint-Staged:
    • Configure a pre-commit hook.
    • It should run tsc (type check) and lint on staged files.
    • Prevent the commit if there are errors.

Level 2: Senior Constraints (Strictness)

  • Absolute Imports: Configure tsconfig.json and Vite to allow imports like @/components/Button instead of ../../components/Button.
  • Strict Null Checks: Ensure strict: true is enabled in tsconfig.json. Go back and fix any errors this causes in your previous modules.
  • Component Documentation: Install Storybook (or Ladle). Create a story for your "Button" and "Card" components. This forces you to isolate your UI from your business logic.

Level 3: Nice to Haves

  • CommitLint: Add commitlint to your Husky hooks to enforce "Conventional Commits" (e.g., blocking "wip" or "fixed stuff" messages).
  • GitHub Actions: Create a .github/workflows/ci.yml file. It should run npm install, npm run lint, and npm run test on every push to main.

3. Test on your own

Verify your Workflow

  • The "Bad Commit" Test: Add a console.log('test') to a file. Try to commit it. Does Husky stop you?
  • The "Bad Type" Test: Introduce a type error (e.g., const x: number = "string"). Try to commit. Does tsc stop you?
